# .gitignore Best Practices

## Start With a Template
Don't write `.gitignore` rules from scratch — use GitHub's maintained templates for your language/framework:
👉 [github.com/github/gitignore](https://github.com/github/gitignore)

Or generate one instantly at [gitignore.io](https://www.toptal.com/developers/gitignore) by specifying your stack (e.g., `Node,React,VSCode,macOS`).

## What Should ALWAYS Be Ignored

### 1. Dependencies (regenerable from a manifest file)
```gitignore
node_modules/
vendor/
.venv/
__pycache__/
```

### 2. Build Output
```gitignore
dist/
build/
out/
*.class
*.o
```

### 3. Environment Variables & Secrets
```gitignore
.env
.env.local
.env.*.local
*.pem
*.key
secrets.json
```

### 4. Logs and Temporary Files
```gitignore
*.log
*.tmp
npm-debug.log*
yarn-error.log*
```

### 5. OS-Specific Files
```gitignore
.DS_Store        # macOS
Thumbs.db        # Windows
Desktop.ini      # Windows
```

### 6. IDE/Editor Files
```gitignore
.vscode/
.idea/
*.swp
*.swo
```

### 7. Test Coverage & Cache
```gitignore
coverage/
.cache/
.pytest_cache/
```

## Repo-Specific vs Global `.gitignore`

**Repo-specific** (`.gitignore` in project root): project-specific artifacts (`node_modules/`, `dist/`).

**Global** (applies to ALL your repos, configured once):
```bash
git config --global core.excludesfile ~/.gitignore_global
```
Use this for personal editor/OS files that shouldn't clutter every project's `.gitignore` (e.g., `.DS_Store`, `.vscode/`).

## Common Mistakes

### ❌ Committing then Ignoring
Adding a file to `.gitignore` **after** it's already tracked does nothing — you must explicitly untrack it:
```bash
git rm --cached path/to/file
git commit -m "Stop tracking file"
```

### ❌ Ignoring Files Needed for the Build
Don't accidentally ignore critical config files (e.g., `.env.example`, `package-lock.json`) that others need to set up the project correctly.

### ❌ Overly Broad Patterns
```gitignore
# Too broad — might accidentally ignore legitimate source files
*.js
```
Be specific:
```gitignore
dist/*.js
```

### ❌ Forgetting Secrets Were Already Committed
If a secret was committed **before** adding it to `.gitignore`, it still exists in Git history. `.gitignore` only prevents *future* tracking. You must:
1. Rotate/invalidate the exposed secret immediately.
2. Use `git filter-repo` or BFG Repo-Cleaner to purge it from history if needed.

## Checking What's Ignored
```bash
git status --ignored     # Show ignored files
git check-ignore -v file # Explain WHY a file is ignored (which rule matched)
```

## Summary
A well-maintained `.gitignore` keeps your repository clean and secure — start from a language-specific template, always exclude secrets/dependencies/build output, and remember it only affects files not yet tracked.
